软件测试

软测全景

如果把软件开发比作建造一栋大楼，那软件测试就是对这栋大楼进行质量验证、风险控制和缺陷预防的全过程。

我刚开始学习的时候认为：

测试 = 找Bug

实际上，这是对软件测试最常见的误解。

国际软件测试资格认证组织（ISTQB）对测试的定义如下：

测试是为了发现缺陷、评估质量并降低风险而进行的一系列验证和确认活动。

现代软件测试的核心目标已经从“找Bug”发展为：

  • 保证产品质量
  • 降低业务风险
  • 提高交付效率
  • 保障用户体验
  • 支撑持续交付

一、软件测试是什么

1. 软件质量的本质

软件质量不是单纯的Bug数量少，而是功能正确，性能稳定，安全可靠，易于维护，用户满意多方面因素共同构成的。

例如：

一个电商系统功能完全正常，但页面打开10秒就高并发崩溃且用户密码明文存储，仍然是低质量系统。

2. 测试的本质（double V）

测试本质上是在回答两个问题：

验证（Verification）

做得对不对？

验证：

软件是否按照需求实现

例如：

需求：

用户密码长度必须8~20位

测试：

  • 7位
  • 8位
  • 20位
  • 21位

验证实现是否正确。

确认（Validation）

做的是不是用户真正需要的？

例如：

产品设计：

用户注册必须填写15项信息

功能没问题。

但用户增长特别缓慢。

这属于：

  • 功能正确
  • 产品失败

这是确认层面的问题。

二、软件测试的发展历程

第一阶段：调试时代

1970年前后

程序员自己测试。

特点：

  • 无专职测试
  • 边开发边修复

第二阶段：独立测试

1980~2000

出现专门测试团队。

特点：

  • 开发写代码
  • 测试找Bug

形成经典模式：

Dev → Test → Release

第三阶段：自动化测试

2000~2015

互联网高速发展。

人工回归成本过高。

出现：

  • Selenium
  • QTP
  • Robot Framework

自动化测试开始普及。

第四阶段：DevOps时代

2015以后

强调：

  • 测试左移
  • 持续集成
  • 持续交付

测试不再是最后一道工序。

而是：

全生命周期质量保障

第五阶段：AI测试时代

当前趋势：

  • AI生成测试用例
  • AI生成自动化脚本
  • AI辅助缺陷分析
  • AI风险预测

但：

AI不能替代测试工程师。

因为测试本质是：

  • 业务理解
  • 风险分析
  • 质量决策

这些仍然依赖人的判断。

三、软件测试分类体系

软件测试可以从多个维度划分，在后续章节进行说明。

四、按照是否查看代码划分

1. 黑盒测试

最常见

特点：

  • 不关心代码
  • 只关注输入输出

例如：

登录功能：

输入：

  • 用户名
  • 密码

检查：

  • 登录成功
  • 登录失败

优点

  • 接近用户视角
  • 学习成本低

缺点

  • 覆盖率有限

2. 白盒测试

关注：

  • 代码结构
  • 分支逻辑
  • 路径覆盖

例如：

测试：

  • age=18
  • age=17

确保两个分支都执行。

常见覆盖率

  • 语句覆盖
  • 分支覆盖
  • 条件覆盖
  • 路径覆盖

3. 灰盒测试

介于两者之间。

测试人员：

  • 知道系统架构
  • 了解数据库
  • 不直接阅读源码

互联网公司最常见。

五、按照测试阶段划分

1. 单元测试（Unit Test）

最小测试单位。

例如：

验证：

通常由开发完成。

工具：

  • JUnit
  • TestNG
  • pytest

2. 集成测试（Integration Test）

验证模块间协作。

例如：

订单系统

调用：

  • 用户服务
  • 商品服务
  • 库存服务

检查整体流程。

3. 系统测试（System Test）

测试完整系统。

例如：

从：

登录

到：

下单

到：

支付

到：

发货

全链路验证。

4. 验收测试（UAT）

User Acceptance Test

用户验收。

确认：

产品满足业务需求

通常由：

  • 客户
  • 产品经理
  • 业务方

参与。

六、按照测试目标划分

功能测试

验证：

功能是否符合需求。

占测试工作的基础部分。

性能测试

验证：

系统运行效率。

关注：

  • 响应时间
  • TPS
  • QPS
  • CPU
  • 内存

工具：

  • JMeter
  • LoadRunner

性能测试分类

负载测试

系统正常压力。

例如：

1000用户同时在线。

压力测试

突破极限。

例如：

10000用户同时访问。

稳定性测试

长时间运行。

例如：

72小时压测。

容量测试

确定最大承载能力。

七、安全测试

目标：

发现安全漏洞。

常见漏洞

SQL注入

错误：

输入：

可能绕过登录。

XSS

跨站脚本攻击。

例如：

输入：

页面执行脚本。

CSRF

跨站请求伪造。

权限绕过

普通用户访问管理员接口。

参考安全体系常以 OWASP Foundation 发布的风险清单为基础。

八、兼容性测试

验证：

软件在不同环境运行情况。

浏览器兼容

例如：

  • Google Chrome
  • Microsoft Edge
  • Mozilla Firefox

操作系统兼容

例如：

  • Windows
  • Linux
  • macOS

移动设备兼容

例如：

  • Android
  • iOS

因为：

系统越来越多采用前后端分离。

接口测试验证：

  • 参数正确
  • 返回正确
  • 数据正确
  • 权限正确

例如：

常见工具：

  • Postman
  • Apifox
  • JMeter

十、自动化测试

目的：

把重复工作交给机器。

适合自动化的场景

高频回归

例如：

登录功能。

每个版本都要测。

数据驱动场景

例如：

1000组测试数据。

人工成本过高。

CI/CD场景

代码提交自动执行。

不适合自动化的场景

  • UI频繁变化
  • 一次性需求
  • 探索性测试

工具体系

Web自动化

  • Selenium
  • Playwright
  • Cypress

App自动化

  • Appium

十二、缺陷（Bug）管理

Bug生命周期：

可能出现：

高质量Bug报告包含：

标题

准确描述问题。

环境

测试环境信息。

前置条件

出现问题前状态。

重现步骤

一步一步操作。

实际结果

发生了什么。

预期结果

应该发生什么。

证据

日志、截图、视频。

十三、测试管理

大型项目中非常重要。

包括：

测试计划

测什么。

测试策略

怎么测。

风险评估

优先测什么。

测试报告

质量是否达标。

常见指标：

  • 用例通过率
  • Bug数量
  • Bug修复率
  • 自动化覆盖率
  • 缺陷密度

十四、测试工程师成长路线

初级测试工程师

掌握：

  • 功能测试
  • 用例设计
  • Bug管理

中级测试工程师

掌握：

  • 接口测试
  • 数据库
  • Linux
  • 抓包

高级测试工程师

掌握：

  • 自动化测试
  • 性能测试
  • CI/CD

测试开发工程师（SDET）

掌握：

  • Java/Python/Go
  • 测试框架开发
  • 平台建设

测试架构师

负责：

  • 企业质量体系
  • 自动化平台
  • 测试战略
